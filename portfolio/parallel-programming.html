Just last winter quarter, I learned parallel programming, and I really like how much better of a programmer I became because of it. I'm now able to code really fast programs using OpenMP, R Snow, CUDA, and Thrust. As someone who really loves to program, it makes me really happy.

For our final project, I worked with three of my peers in parallelizing the combn() function from the R CRAN package 'combinat'. This function is pretty straightforward. As per its official documentation, the combn() function: "Generate all combinations of the elements of x taken m at a time." Click here for a more detailed description.

The goal of the project is to write a new version of the function that works exactly like the original version (handles the same types of inputs, produces the same results for same inputs, etc.), except the new one is parallelized using OpenMP, R Snow, CUDA, and/or Thrust (we choose three, R Snow is required). Even though the combn() function seems straightforward, this project was quite challenging because we had to think of ways that doesn't simply parallelize the implementation, but actually makes the program run faster. It's quite easy to convert the original code into a parallelize version, but it's another thing to actually make considerable speed improvements. There are many factors to consider depending on the platform being used, such as arranging good cache behavior, managing shared memory usage, etc. Another challenging was aspect was handling the input and output for the function. The codes for OpenMP, CUDA, and Thrust need to be written with C++, and going from handling R data types to C++ data types definitely involved quite some work.

It was a challenging project, but overall I was very happy with the speed improvements that my grouped achieved, except for our R Snow implementation, which for a reason I haven't figured out, achieved the exact opposite of our goal. The following charts show the improvements in speeds for inputs up to millions in size.
